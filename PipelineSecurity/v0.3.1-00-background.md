## Purpose

This draft describes high-level options to sign software artifacts, preserve release integrity, and enforce trust at deployment time.

It is written for a mixed environment with:
- Open-source dependencies and internal software.
- Linux packages, Windows binaries, and container images.
- Harbor and/or Nexus as artifact repositories.
- Huawei Cloud private or dedicated cloud services for key custody and signing support.

## Goals

- Ensure artifacts are signed before release.
- Prove that artifacts passed required security checks.
- Keep release repositories immutable.
- Replicate artifacts safely into more secure environments.
- Enforce runtime controls so only trusted artifacts can be deployed.
- Keep private signing keys out of CI/CD workers.

## Integration position for Huawei private cloud

Huawei DEW provides KMS and Dedicated HSM services, and the Dedicated HSM feature set includes signing data and code, signature verification, secure key storage, PKCS#11, and CSP support. That makes Huawei private cloud a suitable trust anchor for signing services hosted on VMs or containers inside the same Huawei environment.

The main design decision is whether the signing tool can use Huawei key custody directly or should use an internal adapter service:
- Cosign supports KMS-backed signing, but the documented built-in providers are AWS KMS, GCP KMS, Azure Key Vault, HashiCorp Vault, and Kubernetes Secrets, so Huawei KMS is not a documented native provider in the current source set.
- Notation is designed to work with built-in providers and external plugins, including integrations for remote KMS services and PKCS#11-based devices.
- SignServer CE can be used as a central signing component, but the preferred Huawei-aligned pattern in this draft is to place it behind Huawei-hosted key custody or a signing API rather than expose raw keys to CI/CD.

## Reference architecture principles

1. Build first, sign last.
2. Sign the immutable artifact digest, not only a mutable tag or filename.
3. Store release evidence as signed metadata or attestations.
4. Promote by digest from staging to release.
5. Re-verify after transfer into a more secure environment.
6. Enforce verification at admission or install time.
7. Keep signing keys in KMS, HSM, or a dedicated signing service.

## Initial option set

| Option | Best fit | Core idea | Main free tools | Huawei-aligned key pattern |
|---|---|---|---|---|
| Harbor-first | Container-heavy environment | Use Harbor as the primary OCI trust and promotion point | Harbor, Cosign, Notation, in-toto | Internal signing service backed by DEW KMS or Dedicated HSM |
| Nexus-first | Mixed packages and binaries | Use Nexus repositories for staged promotion and signed package release | Nexus OSS, SignServer CE, rpmsign, dpkg-sig, debsign, SignTool | Central signing service or SignServer CE backed by Huawei-controlled keys |
| Huawei DEW-backed hybrid | Need centralized key custody | Put private keys behind DEW KMS or Dedicated HSM and expose signing as a service | DEW KMS, Harbor or Nexus, Cosign or Notation, SignServer CE | DEW KMS for API-based signing or Dedicated HSM for PKCS#11/CSP patterns |

## Tool-to-KMS integration view

| Tool | Direct Huawei KMS integration | Practical integration path in Huawei private cloud |
|---|---|---|
| Cosign | Not documented as a native Huawei KMS provider in the current Cosign KMS list | Use an internal signing service backed by DEW KMS, or use a Huawei-hosted key service that exposes a compatible interface for signing by digest |
| Notation | Feasible through its plugin model for remote KMS services or PKCS#11-based providers | Build or adopt a Huawei KMS or Dedicated HSM signing plugin and keep trust policy in Notation |
| SignServer CE | Use as a central signing system hosted in Huawei private cloud | Back it with Huawei-controlled key custody or place it behind an internal signing API; Dedicated HSM is the cleaner path when PKCS#11 or CSP-compatible integrations are needed |
| rpmsign / debsign / SignTool | Usually expect native key or certificate workflows | Use a central signing service, or HSM-backed workflows where the packaging or code-signing stack supports external providers |

## Recommended Huawei-hosted patterns

### Pattern A: DEW KMS signing service

Use DEW KMS as the cryptographic root and expose a small internal signing API running on Huawei private cloud VMs or containers.

Good fit for:
- Cosign-style digest signing where direct Huawei KMS support is not native.
- Centralized approval and auditing.
- Strong separation between build runners and key custody.

### Pattern B: Dedicated HSM adapter pattern

Use Huawei Dedicated HSM when a signing component or plugin can work through PKCS#11 or CSP-oriented integration patterns.

Good fit for:
- Notation plugin development.
- Code-signing workflows that benefit from hardware-backed key custody.
- Higher-assurance isolation and future commercial PKI integration.

### Pattern C: Repository-centric release control

Use Harbor or Nexus as the release authority while the actual signing operation happens through a Huawei-backed signing service.

Good fit for:
- Teams that want one consistent promotion process across containers, packages, and binaries.
- Mixed free and commercial tool usage.

## On-premise buildability

The preferred delivery model remains on-premise or private-cloud hosted signing and release control.

| Capability | Open-source-first path | Commercial examples | Notes |
|---|---|---|---|
| OCI image signing | Cosign, Notation, Harbor | Venafi CodeSign Protect, DigiCert Software Trust Manager, SignPath | Open-source is feasible for OCI artifacts; Huawei key custody usually needs an adapter or plugin path. |
| Package and binary signing | SignServer CE, rpmsign, dpkg-sig, debsign, SignTool | Venafi, DigiCert, SignPath | Open-source is feasible, but Windows and enterprise workflow governance are often easier with commercial tooling. |
| Key custody | Huawei DEW KMS, Dedicated HSM | Vendor-managed signing platforms backed by HSMs | Huawei-native custody is suitable for private-cloud hosted services. |
| Runtime enforcement | Sigstore policy-controller, signed repo verification | Commercial policy and trust platforms | Kubernetes enforcement is strong with open-source; server-side controls may need OS-native controls as well. |

## Release metadata expectations

The release metadata should be bound to the artifact digest and should include:
- Build pipeline ID.
- Source commit or tag.
- Builder identity.
- SBOM reference and hash.
- SCA result.
- SAST result.
- DAST result if applicable.
- Manual approval or change record.
- Release date and target environment.
- Expiry or review window for the release decision.

## High-level risks and controls

| Risk | High-level control |
|---|---|
| Artifact changed after approval | Immutable release repositories, no redeploy, promotion by digest |
| Scan evidence lost or disconnected from artifact | Signed attestations tied to artifact digest |
| Unsigned content deployed | Admission or install-time verification |
| Key compromise on build workers | Keep keys in KMS, HSM, or central signing service |
| Replication tampering | Re-verify digest, signature, and attestation in destination zone |
| Non-compliant dependency introduced | Staging and policy gates before release |


## Published guidance status

| Product or area | Official published guidance found | What it covers | Custom integration likely required |
|---|---|---|---|
| Huawei DEW KMS | Yes | General KMS usage patterns and service integration concepts | No for KMS itself, yes for tool-specific signing adapters |
| Huawei Dedicated HSM | Yes | Dedicated HSM capabilities, network deployment, API and SDK access, and hardware-backed key use | Usually yes for product-specific signing workflows |
| Huawei SWR or CCE image signing and verification | Yes | Image signing with DEW-backed keys and signature verification add-on guidance in Huawei container services | Less custom work if you stay inside Huawei-native container patterns |
| Harbor with Cosign or Notation | Yes | OCI artifact signing and verification workflows inside Harbor | Huawei-specific key integration still needs local design |
| Cosign with Huawei KMS | No direct official end-to-end guide found | Cosign KMS model is documented, but Huawei KMS is not listed as a native provider in the current source set | Yes |
| Notation with Huawei KMS or HSM | No Huawei-specific end-to-end guide found | Notation plugin extensibility is documented | Yes, likely via plugin or adapter |
| SignServer CE with Huawei KMS | No official Huawei-specific guide found | SignServer CE is documented as open-source signing software | Yes, likely via broker, adapter, or HSM-backed integration |
| Nexus with signed release controls | Yes for repository controls | Immutability pattern, policy controls, and SBOM handling are documented | Yes for Huawei-specific signing path |
