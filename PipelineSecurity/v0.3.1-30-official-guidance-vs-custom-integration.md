## Purpose

This document separates areas where there is published product guidance from areas where custom integration work is still likely required in a Huawei private cloud software-signing architecture.

## Design goal

The target capability is enterprise artifact signing that can be built and operated on premises or in Huawei private cloud, using open-source tools where feasible and commercial tooling where governance, platform support, or HSM integration makes that more practical.

## Interpretation rule

Use official guidance when the vendor or project documents the feature or workflow directly.

Treat the solution as custom integration when:
- There is no Huawei-specific implementation guide for the tool.
- The tool supports plugins or external providers, but no Huawei plugin is documented.
- The repository or runtime control is documented, but the key-custody integration is not.

## Buildable open-source baseline

A realistic on-premise baseline for enterprise artifact signing is:
- Harbor for OCI storage and promotion.
- Nexus OSS where mixed package formats need repository management.
- Cosign or Notation for OCI signatures.
- SignServer CE plus rpmsign, dpkg-sig or debsign, and SignTool for packages and binaries.
- in-toto attestations for release metadata.
- Huawei DEW KMS or Dedicated HSM for key custody behind a broker or adapter.
- Sigstore policy-controller for Kubernetes runtime enforcement.

This baseline is buildable with open-source components, but Huawei-specific key integration for Cosign, Notation, and SignServer CE should be planned as an internal engineering stream rather than assumed to exist as a ready-made integration.

## Where open source is feasible

Open source is strongest for:
- OCI image and artifact signing.
- Attestation and release metadata.
- Kubernetes verification gates.
- Repository-side immutability and promotion control.

Open source is more demanding for:
- Centralized signing workflow governance.
- Windows enterprise code-signing at scale.
- Deep HSM integration with minimal custom engineering.
- Multi-team approval flows, audit controls, and delegated signing policies.

## Recommended architectural stance

### Use published guidance as-is

- Huawei DEW KMS or Dedicated HSM setup.
- Huawei image signing and image verification features where you stay inside Huawei-native container services.
- Harbor signing and verification workflows.
- Nexus immutability and repository policy controls.
- Sigstore policy-controller for Kubernetes enforcement.

### Engineer custom integration

- Cosign signing through a Huawei-aware broker or signing API.
- Notation plugin or adapter for Huawei key custody.
- SignServer CE integration with Huawei-managed key storage.
- Unified attestation and metadata transport across Harbor and Nexus.

## Recommended standards language

Use wording such as:
- Official product guidance shall be followed where available.
- Where no Huawei-specific supported integration is published, the organization shall use an internal broker, adapter, or plugin pattern with defined ownership, testing, and audit logging.
- Private signing keys shall remain in Huawei DEW KMS or Dedicated HSM, or another approved hardware-backed service.
